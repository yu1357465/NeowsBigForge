# NeowsBigForge V2.0 架构说明书
**杀戮尖塔2 (Slay the Spire 2) 定性构筑沙盘系统**

## 1. 核心设计哲学 (Design Philosophy)

NeowsBigForge V2.0 脱胎于传统的“数据计算器”，转型为真正的“定性推演沙盘”。在架构设计上，我们遵循以下三个核心原则：

* **定性优于定量 (Qualitative > Quantitative)：** 在肉鸽卡牌游戏中，宏观的“端口平衡”（如是否缺少过渡防御、润滑运转）的权重，远远高于微观的“单卡上手概率”。系统通过抽象的标签系统，将玩家从繁杂的数字中解放出来。
* **物理直觉与逻辑严密 (Physical Intuition)：** 卡组的能量负载计算回归最底层的物理模型。不排除基础牌，不忽略诅咒牌。任何占用抽牌位的卡牌都被严格计入厚度分母，真实反映删牌或加牌对卡组流畅度的绝对影响。
* **做减法 (Subtractive Design)：** 摒弃消耗算力且实战指导意义极低的超几何分布算法，砍掉多余的视觉噪音，将算力和屏幕空间100%聚焦于核心的“协同雷达”与“推演台”。

## 2. 系统层级与数据流转 (System Architecture)

本系统采用纯前端 (Pure Front-End) 无服务器架构，依托 GitHub Pages 部署，极大地保证了响应速度与数据隐私。

系统的存储层被严格划分为三个物理与逻辑层级：

1.  **静态只读层 (Static Database)：** `STS2_Card_Database_ZHS.json`。包含所有卡牌的基础属性（费用、描述、职业）。系统启动时通过 Fetch API 异步拉取，并在 URL 尾部添加时间戳以击穿浏览器缓存。
2.  **持久化记忆层 (Hard Drive / LocalStorage)：** * `SpireV2_Dictionary`：存储玩家自定义的卡牌鉴定数据（梯度、标签）。
    * `sts2_v2_save_[职业]_[槽位]`：存储各职业独立隔离的存档卡组。
    * `SpireV2_SlotNames`：存储用户自定义的方案名称。
3.  **沙盘工作区 (Working Memory / RAM)：** `myDeck` 与 `myDrafts` 数组。UI 面板的所有增删改查仅在内存中进行，实现“What-If”的无损推演。

## 3. 核心引擎解析 (Core Engines)

### 3.1 命名空间隔离与读写分离引擎
* **底层逻辑 (Mechanism)：** 彻底切断了 UI 操作与硬盘存档之间的自动绑定。引入了基于顶级职业前缀的命名空间（Namespace）隔离。
* **实现方式：** * 在切换职业时，系统会重置工作区，并挂载对应的独立存档槽位（如从 `regent_slot1` 切换到 `ironclad_slot1`）。
    * 修改只停留在 UI 层，只有当用户明确点击“覆盖存档”时，才会执行 `commitSaveToDisk` 将内存数组序列化写入 LocalStorage。点击“撤销”则从 LocalStorage 强行覆盖当前内存。

### 3.2 动态负载与协同推演引擎
* **能量负载算法：** 采用 `(总费用 / 真实卡牌总数) * 5` 的公式推算首回合期望耗费。计算 Delta EV（增量期望）时，精准捕捉新加入卡牌对整个卡池密度的稀释或加重效应。
* **协同分析评分系统：** 对候选卡牌进行多维度计分。如果卡组极度缺乏某种标签（如终端输出），该标签卡牌将获得“雪中送炭”的高额权重加成；反之，若卡组某种标签密度超过 35%，则触发“冗余”扣分惩罚。

### 3.3 视觉与渲染引擎 (CSS Grid & Tooltip)
* **反直觉提示 (Counter-Intuitive)：** 为了实现丝滑的悬浮标签，最佳方案不是使用 JavaScript 去监听鼠标事件和计算坐标，而是利用 CSS 的 `:hover` 伪类和绝对定位。
* **布局逻辑：** 中间的卡组展示区采用 CSS Grid 网格布局，通过 `repeat(auto-fill, minmax(130px, 1fr))` 实现自适应的瀑布流。同时，利用 `!important` 强制解除外层容器的 `overflow: hidden` 和层级压制，确保高级深色悬浮提示框能够无视 DOM 结构层级，在最高维度向下弹出，彻底消除了界面遮挡与截断问题。

## 4. 扩展性预留 (Scalability)

当前 V2.0 版本已为《杀戮尖塔2》的全面发售做好了底层准备：
1.  **辞典热更新支持：** 标签系统已通过脚本接口暴露，未来可一键注入其他社区大佬的最新职业攻略，实现零代码维护。
2.  **词条解析器解耦：** `parseCardStats` 引擎已具备对“费用变为 X”、“费用加 1”等相对值修饰词的拦截与兼容能力，足以应对未来卡牌描述的复杂化迭代。
